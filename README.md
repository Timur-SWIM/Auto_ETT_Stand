# Стенд на MDR32F9Q2I

Проект содержит прошивку для учебного стенда на микроконтроллере **Milandr MDR32F9Q2I**. В текущем состоянии прошивка объединяет три задачи:

- измерение температуры через АЦП и периодическую отправку телеметрии по USB CDC;
- формирование управляющего уровня для внешнего генератора через `DAC + DMA + TIMER2`;
- управление цифровым аттенюатором через линии `PORTA[5:0]`.

Проект собирается в **Keil uVision 5** и использует стандартную библиотеку периферии Milandr (`CMSIS` + `SPL`).

## Что делает прошивка

### Измерение температуры

- АЦП работает в циклическом режиме.
- DMA непрерывно заполняет окно из 10 отсчетов.
- Главный цикл усредняет эти 10 значений и переводит код АЦП в градусы Цельсия функцией `ADC_ToTemp()`.
- Раз в период `TIMER1` температура отправляется в USB в виде строки `TEMP: <value> C`.

### ШИМ и PID

- `TIMER3` формирует ШИМ на `PB0`.
- `PID_Update()` пересчитывает скважность в главном цикле.
- Уставка PID сейчас не задается по USB: в коде она зафиксирована как середина диапазона `T_MIN..T_MAX`.

### Генератор на DAC

- Таблица `DAC_DMA_Data[10]` содержит 10 кодов ЦАП.
- DMA для `TIM2` работает в режиме `ping-pong` с двумя буферами `DAC_DMA_Buffer[2][10]`.
- Командный слой изменяет неактивный буфер, а переключение на новые данные происходит в `DMA_IRQHandler()` на безопасной границе DMA-цикла.
- Калибровочная таблица `vco_table[]` связывает частоту генератора в МГц, код DAC и опорный уровень мощности.

### Аттенюатор

- `PORTA[5:0]` используются как 6-битный код управления аттенюатором.
- Команда аттенюатора рассчитывается из требуемой мощности и диапазона, который соответствует подготовленной DAC-таблице.

### USB-интерфейс

- USB работает как `CDC Virtual COM Port`.
- Прием данных идет в кольцевой буфер.
- Парсер принимает только команды, оформленные как кадры `"<...>"`.
- В одном USB-кадре можно передать несколько команд, разделяя их запятой.

## Формат команд

Поддерживаемый синтаксис команд соответствует реализации в [src/sys_command.c](/C:/Users/User/Documents/Milandr/MiTestExample/Stand/src/sys_command.c):

```text
<G.SET.1800>
<G.SPAN.1500.2400>
<G.TABLE.1500.1600.1700.1800>
<G.OFF>
<A.SET.315>
<A.READ>
<I>
<G.SET.1800,A.SET.315>
```

### Команды генератора

- `G.SET.<freq_mhz>`: заполнить весь DAC-буфер одним значением частоты.
- `G.SPAN.<f_start>.<f_stop>`: построить линейный диапазон из 10 точек.
- `G.TABLE.<f1>...<fN>`: записать набор точек вручную; оставшиеся элементы будут заполнены последним значением.
- `G.OFF`: записать нули в DAC-буфер.

### Команды аттенюатора

- `A.SET.<p_out>`: выставить код аттенюатора для требуемой выходной мощности.
- `A.READ`: заглушка, сейчас только возвращает код успешного выполнения внутри прошивки.

### Информационная команда

- `I`: отправляет строку `MDR32_ETT_Stand_Board`.

## Ключевые связи по периферии

- `PB0`: PWM-выход (`TIMER3`, channel 1).
- `PB7`: светодиод-индикатор, переключается в обработчике `TIMER1`.
- `PORTE0`: аналоговый выход `DAC2`.
- `PORTA[5:0]`: линии управления аттенюатором.
- `USB CDC`: прием команд и отправка телеметрии.
- `ADC2`: вход измерительного канала температуры.

## Последовательность запуска

Главный цикл в [src/main.c](/C:/Users/User/Documents/Milandr/MiTestExample/Stand/src/main.c) делает следующее:

1. настраивает тактирование;
2. инициализирует DMA и АЦП;
3. поднимает PWM, DAC, таймеры и порты;
4. запускает USB CDC;
5. в бесконечном цикле:
   - берет среднее значение АЦП;
   - обновляет PID;
   - извлекает команду из USB-потока;
   - по флагу от `TIMER1` отправляет температуру.

## Сборка

### Требования

- Keil uVision 5;
- пакет поддержки для `MDR32F9Q2I`;
- подключенная отладочная плата или целевой стенд.

### Шаги

1. Откройте `stand.uvprojx` в Keil.
2. Проверьте выбранное устройство `MDR32F9Q2I`.
3. Соберите проект.
4. Запишите прошивку в микроконтроллер.
5. Подключите USB и откройте виртуальный COM-порт в терминале.

## Примечания по коду

- Обновление таблицы DAC сделано через двойную буферизацию, чтобы не менять данные, из которых DMA читает прямо сейчас.
- Парсер USB специально хранит неполный хвост кадра между вызовами `extract_USB_command()`, поэтому команды могут приходить фрагментами.
- Ряд функций отладки (`USB_PrintDebug`) сейчас активен без условной компиляции и сразу отправляет данные в CDC-канал.
